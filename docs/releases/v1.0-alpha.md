# Release Notes: Causa Core v1.0-alpha

## What v1.0-alpha includes

- **Data Models**: Python dataclasses for `AnchorCase`, `UISnapshot`, `LLMProposal`, `HumanApproval`, `ExecutionEvent`, `OutcomeEvidence`, `ReplayPolicy`, `ReplayDecision`, `AnchorRecord`, and `AuditSummary`.
- **Decision Logic**: A pure deterministic function (`determine_replay_decision` and `determine_replay_decision_from_record`) that assesses replay eligibility based on strict matching rules and risk thresholds.
- **Audit Summary**: An informational layer (`build_audit_summary`) providing human-readable explanations of replay decisions.
- **Storage Abstraction**: The `CausaRepository` abstract base class establishing a storage-agnostic interface.
- **Test Infrastructure**: `InMemoryCausaRepository` alongside comprehensive, non-executing, deterministic end-to-end simulation tests spanning the entire decision pipeline.

## What it explicitly does not include

- OS execution
- Autonomous replay
- Accessibility API calls
- Dify integration
- Sudo or privilege escalation
- Network calls
- Destructive operations
- Production database usage
- Supabase integration
- PostgreSQL integration

## Architecture flow

`AnchorRecord` → `InMemoryCausaRepository` → `determine_replay_decision_from_record` → `ReplayDecision` → `build_audit_summary` → `AuditSummary`

## Safety constraints

The `causa_core` library is strictly a logic and data definition module. It makes decisions and generates audits, but fundamentally cannot mutate host state, escalate privileges, or communicate over external networks. The entire suite runs deterministically without invoking actual OS actions.

## Storage direction

Causa Core is explicitly storage-agnostic. The `InMemoryCausaRepository` is intended strictly for tests and local simulation. The core decision logic (`decision.py`) must remain completely independent from storage implementations.

## Known limitations

- `get_anchor_record` via `InMemoryCausaRepository` currently fetches the first matching related records.
- Partial matches strictly reject replays and require LLM reasoning; no fuzzy matching is natively supported yet.

## Next likely milestone

The next milestone is to build a SQLite local adapter for persistence while maintaining the strict decoupling from the core decision logic.
